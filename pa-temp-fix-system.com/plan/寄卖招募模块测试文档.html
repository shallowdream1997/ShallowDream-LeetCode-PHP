1. 测试概述 概述

本文档涵盖 寄卖招募模块 全部API的测试用例，包括：

  • 功能测试 — 每个API的正常流程与参数组合验证
  • 集成测试 — 跨表数据一致性、状态流转正确性
  • 并发测试 — Redis锁保护下的竞争池原子性、并发加入/撤销
  • 边界测试 — 竞争池满员、重复操作、空列表、不存在记录等

测试范围：pa-scms-service 模块的 /v1/cart/* 和 /v1/list/* 共 9个API。

    TC-01：加入招募车 — 正常流程

    P0 POST

    API: POST /cart/join

    前置条件

    • 招募清单 SC20260601001 状态为「招募中」(20)，竞争池人数=2
    • 供应商 1003（丙供应链）尚未加入该清单

    请求

    { "recruitNo": "SC20260601001", "supplierId": 1003, "groupId": 2003 }

    后台处理步骤

    1. 根据 recruitNo 查询 recruit_list → id=1, list_status=20, supplier_apply_count=2
    2. 申请Redis锁 key="consignment:recruit:lock:1" → 抢锁成功
    3. 查询是否已重复加入 SELECT * FROM recruit_apply WHERE recruit_id=1 AND supplier_id=1003 AND is_deleted=0 → 无记录
    4. 乐观锁更新竞争池 UPDATE recruit_list SET supplier_apply_count=supplier_apply_count+1 WHERE id=1 AND supplier_apply_count < 5 → 影响1行 (2→3)
    5. 新增申请记录 → INSERT INTO recruit_apply (recruit_id, supplier_id, apply_status=10, ...)
    6. 释放Redis锁

    数据走向

    ┌─ recruit_list ──────────────────┐ ┌─ recruit_apply ──────────────────┐ │ id=1 │ │ id=3 ← NEW │ │ supplier_apply_count: 2 → 3 │ │ recruit_id: 1 │ │ list_status: 20 (招募中) │ │ supplier_id: 1003 │ └───────────────────────────────────────────┘ │ supplier_name: 丙供应链公司 │ │ apply_status: 10 (已加入) │ │ ce_bill_no: null │ └────────────────────────────────────────┘
    验证点
    ✅ 接口返回 {"data": {"value": true}}
    ✅ recruit_list.supplier_apply_count = 3（比之前+1）
    ✅ recruit_apply 新增一条记录，apply_status=10
    ✅ 再次调用同一接口应返回失败（重复加入检测）

    TC-02：撤回招募车 — 正常流程

    P0 POST

    API: POST /cart/withdraw

    前置条件

    • 供应商 1001（甲贸易）已加入清单 SC20260601001，apply_id=1, status=10
    • 竞争池人数=3（含甲）

    请求

    { "recruitNo": "SC20260601001", "supplierId": 1001, "groupId": 2001 }

    后台处理步骤

    1. 根据 recruitNo 查询 recruit_list → id=1
    2. 根据 recruitId+supplierId 查询申请 → apply_id=1, status=10
    3. 申请Redis锁 key="consignment:recruit:lock:1"
    4. 物理删除申请：DELETE FROM scms_consignment_recruit_apply WHERE id=1
    5. 竞争池降级：UPDATE recruit_list SET supplier_apply_count = IFNULL(supplier_apply_count, 0) - 1 WHERE id=1 → 3→2
    6. 释放Redis锁

    数据走向

    ┌─ recruit_apply ──────────────────┐ ┌─ recruit_list ──────────────────┐ │ id=1 (DELETED) ░░░░░░░░░░░░░░░░░░░│ │ id=1 │ │ recruit_id=1 ░░░░░░░░░░░░░░░░░░░│──物理删除──│ supplier_apply_count: 3 → 2│ │ supplier_id=1001 ░░░░░░░░░░░░░░░│ │ │ └────────────────────────────────────────┘ └────────────────────────────────────┘
    验证点
    ✅ 接口返回 {"data": {"value": true}}
    ✅ recruit_apply 表中 id=1 记录已被物理删除（不复存在）
    ✅ recruit_list.supplier_apply_count = 2（减少1）
    ✅ 再次调用同一接口应返回"申请不存在"

    TC-05：批量开CE单 — 正常流程

    P0 POST

    API: POST /cart/appendSku

    前置条件

    • 供应商1001对清单1已有申请（apply_status=10, ce_bill_no=null）
    • 供应商1001对清单2已有申请（apply_status=10, ce_bill_no=null）
    • 清单1有30个SKU，清单2有50个SKU

    请求

    { "recruitNoList": ["SC20260601001", "SC20260601002"], "supplierId": 1001, "groupId": 2001 }

    后台处理步骤（每个recruitNo）

    1. 根据 recruitNo 查询 recruit_list
      SELECT * FROM scms_consignment_recruit_list WHERE recruit_no='SC20260601001'
    2. 根据 recruitId + supplierId 查询申请 → 验证 apply_status=10 且 ce_bill_no IS NULL
    3. 获取该清单全部SKU
      SELECT * FROM scms_consignment_recruit_list_sku WHERE recruit_id=1
    4. 收集所有 skuId，调用样品服务 getSampleSkuInfoMapBySkuIdList(skuIds)
    5. 创建CE主表 → 调用 ceBillService.createCeMaster() → 获取 ceBillNo
    6. 为每个SKU创建CE明细 → 循环调用 ceBillService.createCeDetails()
    7. 更新申请记录
      UPDATE scms_consignment_recruit_apply SET apply_status=20, ce_bill_no='CE20260601001', ce_create_time=NOW() WHERE id=apply_id

    数据走向

    ┌─ recruit_apply ──────────────────┐ ┌─ CE主表(外部) ─────────────────┐ │ apply_status: 10 → 20 │ │ ce_bill_no: CE20260601001 │ │ ce_bill_no: null → CE20260601001 │ │ supplier_id: 1001 │ │ ce_create_time: null → NOW │ │ factory_id: 101 │ └────────────────────────────────────────┘ └────────────────────────────────────────┘ ┌─ recruit_list_sku ────────────────┐ ┌─ CE明细表(外部) ────────────────┐ │ sku_id=5001 cost=25.00 │───createCeDetails──│ ce_bill_no: CE20260601001 │ │ sku_id=5002 cost=30.00 │ (循环30次) │ sku_id: 5001 temp_id: xxx cost:25.00 │ │ ...共30条 │ │ ...30条明细 │ └──────────────────────────┘ └──────────────────────────────────────────┘
    验证点
    ✅ 接口返回 {"data": {"value": true}}
    ✅ 两个清单的申请 apply_status 均变为 20（已开CE）
    ✅ 两个申请的 ce_bill_no 和 ce_create_time 均有值
    ✅ CE系统中分别有对应主表和明细记录
    ✅ 清单2（50SKU）应创建50条明细

    边界场景

    #场景预期
    TC-05-1清单列表为空 recruitNoList=[]返回 false
    TC-05-2某个清单已有CE单号跳过该清单，其他正常创建
    TC-05-3某个申请状态不为10跳过该清单，返回原因
    TC-05-4清单不存在跳过，继续处理下一个

    7. 并发测试 并发

    TC-C01：并发加入招募车（竞争池满员）

    P0 并发

    场景描述

    同一时刻，5个不同的寄卖商同时申请加入同一清单（池容量5人，已有2人，剩余3个名额）。

    测试数据

    请求supplier_idgroup_id预期结果
    请求A10052005成功 竞争池 2→3
    请求B10062006成功 竞争池 3→4
    请求C10072007成功 竞争池 4→5
    请求D10082008失败 竞争池已满
    请求E10092009失败 竞争池已满

    并发执行流程

    step 1: 5个请求同时到达，各自申请Redis锁 锁 key = "consignment:recruit:lock:1"（超时3秒） step 2: Redis锁保证同一时间只有一个请求执行（串行化） step 3: 请求A 持有锁 → SELECT supplier_apply_count = 2 → UPDATE SET count+1 WHERE count<5 → 影响1行 → count: 2→3 → INSERT apply_record (supplier_id=1005, ...) → 释放锁 step 4: 请求B 持有锁 → SELECT supplier_apply_count = 3 → UPDATE SET count+1 WHERE count<5 → 影响1行 → count: 3→4 → INSERT apply_record (supplier_id=1006, ...) → 释放锁 step 5: 请求C 持有锁 → SELECT supplier_apply_count = 4 → UPDATE SET count+1 WHERE count<5 → 影响1行 → count: 4→5 → INSERT apply_record (supplier_id=1007, ...) → 释放锁 step 6: 请求D 持有锁 → SELECT supplier_apply_count = 5 → UPDATE SET count+1 WHERE count<5 → 影响0行 → 已满，返回失败 → 释放锁 step 7: 请求E 持有锁（同理，竞争池已满返回失败）
    验证点
    ✅ 前3个请求返回 true，后2个返回 false 或业务异常
    ✅ 最终 supplier_apply_count = 5
    ✅ 第6个请求（无论哪个供应商）均因竞争池已满失败
    ✅ recruit_apply 表仅新增3条记录（1005, 1006, 1007）
    ✅ 不存在重复提交导致的同供应商多条记录

    TC-C02：并发撤销招募车

    P0 并发

    场景描述

    3个已加入的供应商（1001, 1002, 1003）同时撤回同一个清单的招募车。

    并发执行流程

    step 1: 3个请求各自申请Redis锁 key="consignment:recruit:lock:1" step 2: 请求A (supplier_id=1001) 成功获锁 → 物理删除 apply_id=1 → UPDATE recruit_list SET supplier_apply_count = 3-1=2 → 释放锁 step 3: 请求B (supplier_id=1002) 成功获锁 → 物理删除 apply_id=2 → UPDATE recruit_list SET supplier_apply_count = 2-1=1 → 释放锁 step 4: 请求C (supplier_id=1003) 成功获锁 → 物理删除 apply_id=3 → UPDATE recruit_list SET supplier_apply_count = 1-1=0 → 释放锁
    验证点
    ✅ 3个请求均返回 true
    ✅ 最终 supplier_apply_count = 0
    ✅ 全部申请记录被物理删除（不是 is_deleted=1）
    ✅ 再次查询该记录不存在（SELECT * 返回空）

    8. 异常场景与边界测试 异常

    #场景操作预期结果
    E-01 重复加入检测 加入成功后立即再次调用同一接口 返回业务异常或 false，recruit_apply 无重复记录
    E-02 撤回后立即重新加入 撤回招募车后不刷新页面，立即再次加入同一清单 加入成功，撤回是物理删除，不会与旧记录冲突
    E-03 已开CE不可重复开单 对已开CE的申请再次调用 appendSkuToCe / createCe 批量模式跳过该记录；单ID模式返回 success=false
    ce_bill_no 不变，CE主表未重复创建
    E-04 清单不存在 传入不存在的 recruitNo 或 recruitId 返回失败提示："清单不存在" 或对应错误消息
    E-05 清单状态无效 对状态不为"招募中"的清单执行加入/开CE操作 返回失败提示："状态不允许" 或 "当前清单不在招募期"
    E-06 竞争池降级不产生负数 竞争池为0时主动撤销招募车（极端情况） IFNULL(supplier_apply_count, 0) - 1 不会出现负数，MySQL执行为0
    E-07 申请不存在（撤回已撤的） 对已撤回（物理删除）的记录再次调用撤回 提示"申请不存在或已删除"
    E-08 供应商不存在 传入不存在的 supplierId supplierService.getSupplierInfo() 返回空，供应商名称为空字符串或默认值
    E-09 必填参数缺失 请求 DTO 缺少 recruitNo / supplierId 等必填字段 参数校验异常，返回 400 Bad Request 或服务端校验提示
    E-10 Redis锁获取超时 某请求长时间持有锁，其他请求超时（3秒） 超时请求返回失败提示"系统繁忙，请稍后重试"

    时序状态转移验证

    stateDiagram-v2
    [*] --> 已加入: joinCart
    已加入 --> 已开CE: createCe / appendSkuToCe
    已加入 --> [*]: withdrawCart (物理删除)
    已开CE --> 等待评选: 评选池(外部)
    等待评选 --> 分配完成: 评选获胜
    等待评选 --> 放弃: 评选落选
