# 寄卖招募模块测试文档

> 本文档用于寄卖招募模块的功能测试、集成测试和并发测试，包含业务流程总览、各场景操作步骤、数据在核心表中的走向追踪、多寄卖商并发场景模拟，以及页面字段与数据来源的完整映射。

## 1. 业务流程总览

### 1.1 核心数据表

| 表名 | 简称 | 核心作用 | 关键字段 |
|------|------|----------|----------|
| `scms_consignment_recruit_list` | recruit_list | 招募清单主表 | id, recruit_no, factory_id, list_status, supplier_apply_count(竞争池人数) |
| `scms_consignment_recruit_list_sku` | recruit_list_sku | 招募清单SKU明细 | id, recruit_id, sku_id, cost_price |
| `scms_consignment_recruit_apply` | recruit_apply | 招募申请记录表 | id, recruit_id, supplier_id, apply_status, ce_bill_no |
| `scms_consignment_recruit_action_log` | action_log | 操作日志表 | id, recruit_id, apply_id, action, operator_id |

### 1.2 核心业务流程

```mermaid
graph TB
    A[寄卖商登录] --> B[招募清单列表页]
    B --> C{是否同源/竞争中?}
    C -->|是| D[加入招募车<br/>(joinCart)]
    C -->|否| E[展示不可加入]
    
    D --> F{REDIS锁<br/>并发保护}
    F -->|抢锁成功| G[检查竞争池<br/>supplier_apply_count]
    F -->|抢锁失败| H[返回失败]
    G -->|竞争池<5| I[新增申请记录<br/>apply_status=10]
    G -->|竞争池>=5| J[返回已满<br/>不可加入]
    
    I --> K[招募车列表页]
    K --> L{后续操作?}
    L -->|撤销招募车| M[撤回招募车<br/>withdrawCart]
    L -->|开CE单| N[开样品CE单<br/>createCe/appendSkuToCe]
    L -->|查看结果| O[招募结果详情<br/>awardDetail]
    
    M --> P{REDIS锁<br/>并发保护}
    P -->|成功| Q[物理删除申请记录<br/>竞争池-1]
    
    N --> R[创建CE主表<br/>创建CE明细<br/>更新申请状态=20]
```

### 1.3 API 总览

| 接口 | 路径 | 核心功能 |
|------|------|----------|
| 招募清单列表分页 | `POST /v1/list/page` | 查看可加入的招募清单，含统计、筛选 |
| 加入招募车 | `POST /v1/cart/join` | 加入清单（Redis锁保护竞争池） |
| 撤回招募车 | `POST /v1/cart/withdraw` | 撤回申请（Redis锁降级竞争池） |
| 招募车列表分页 | `POST /v1/cart/page` | 查看已加入的申请记录 |
| 招募结果详情 | `POST /v1/cart/awardDetail` | 查看清单各供应商排名详情 |
| 开样品CE单 | `POST /v1/cart/appendSku` | 批量按清单编号开CE单 |
| 按清单ID开CE单 | `POST /v1/cart/createCe` | 按招募清单ID开CE单（对外API） |
| 下拉选项 | `GET /v1/selectItemList` | 枚举下拉：awardResult/listStatus/applyStatus/ceStatus |
| 下载附件 | `POST /v1/download` | 下载招募清单附件 |

## 3. 测试场景与数据走向

### 3.1 场景一：寄卖商加入招募车

**操作说明**：寄卖商丙（supplier_id=1003）在招募清单列表页看到清单SC20260601001，点击"加入招募车"。

**前置条件**：竞争池当前人数=2，池容量5人，尚有3个空位。

**请求参数**：
```json
POST /scms/consignment/recruit/supplier/v1/cart/join
{
  "recruitNo": "SC20260601001",
  "supplierId": 1003,
  "groupId": 2003
}
```

**后台处理流程**：
```
step 1: 根据 recruitNo=SC20260601001 查找 recruit_list
        → SELECT * FROM scms_consignment_recruit_list WHERE recruit_no='SC20260601001'
        → 返回 id=1, list_status=20, supplier_apply_count=2

step 2: 申请Redis锁 key="consignment:recruit:lock:1"（超时3秒）
        → 抢锁成功

step 3: 查询当前供应商是否已加入（防重复）
        → SELECT * FROM scms_consignment_recruit_apply 
          WHERE recruit_id=1 AND supplier_id=1003 AND is_deleted=0
        → 返回空，未重复加入

step 4: 检查竞争池是否已满（乐观锁）
        → UPDATE scms_consignment_recruit_list 
          SET supplier_apply_count = supplier_apply_count + 1 
          WHERE id=1 AND supplier_apply_count < 5
        → 影响行数=1，更新成功（supplier_apply_count 2→3）

step 5: 新增申请记录
        → INSERT INTO scms_consignment_recruit_apply 
          (recruit_id, recruit_no, supplier_id, supplier_name, group_id, 
           apply_status, ...) 
          VALUES (1, 'SC20260601001', 1003, '丙供应链公司', 2003, 10, ...)

step 6: 释放Redis锁

step 7: 记录操作日志（可选，前端页面操作时已去除）
```

**数据走向**：

```
 recruit_list(竞争池)              recruit_apply(申请表)
 ┌──────────────────┐            ┌────────────────────────┐
 │ id=1             │            │ id=3 ← NEW             │
 │ supplier_apply_count: 2→3     │ recruit_id: 1          │
 │                   │            │ supplier_id: 1003      │
 └──────────────────┘            │ supplier_name: 丙供应链│
                                 │ apply_status: 10(已加入)│
                                 │ ce_bill_no: null        │
                                 └────────────────────────┘
```

**验证点**：
1. 接口返回 `{"value": true}`
2. `recruit_list.supplier_apply_count` = 3（增加了1）
3. `recruit_apply` 表新增一条记录，apply_status=10
4. 再次调用同一个接口应返回重复加入异常

### 3.2 场景二：多寄卖商并发加入（竞争池满员）

**操作说明**：模拟多个寄卖商同时加入同一清单，测试竞争池满员时的并发保护。

**模拟方式**：使用压测工具（JMeter/Locust）或手动模拟同时发起4个 `joinCart` 请求，此时竞争池已3人，还剩2个名额。

**测试数据**：

| 请求 | supplier_id | group_id | 预期结果 |
|------|-------------|----------|----------|
| 请求1 | 1005 | 2005 | 成功，竞争池3→4 |
| 请求2 | 1006 | 2006 | 成功，竞争池4→5 |
| 请求3 | 1007 | 2007 | 失败，竞争池已满 |
| 请求4 | 1008 | 2008 | 失败，竞争池已满 |

**后台并发保护流程**：
```
step 1: 4个请求同时到达，各自申请Redis锁

step 2: Redis锁保证同一时间只有一个请求执行（串行化）
        锁key = "consignment:recruit:lock:1"

step 3: 第一个请求（请求A）持有锁
        → 检查竞争池 supplier_apply_count=3 < 5 → OK
        → 乐观锁 UPDATE ... SET supplier_apply_count+1 WHERE supplier_apply_count<5
        → supplier_apply_count: 3→4
        → 新增申请记录

step 4: 第二个请求（请求B）持有锁
        → 检查竞争池 supplier_apply_count=4 < 5 → OK
        → 乐观锁 UPDATE ... SET supplier_apply_count<5
        → supplier_apply_count: 4→5
        → 新增申请记录

step 5: 第三/四个请求（请求C/D）持有锁
        → 检查竞争池 supplier_apply_count=5，不小于5
        → 乐观锁 UPDATE ... SET supplier_apply_count+1 WHERE supplier_apply_count<5
        → 影响行数=0 → 竞争池已满，返回失败
```

**数据走向**：

```
 请求A 成功后     请求B 成功后     请求C 失败后
 recruit_list     recruit_list     recruit_list
 supplier_apply   supplier_apply   supplier_apply
  3 → 4            4 → 5            5 (不变)
```

**验证点**：
1. 前2个请求返回 `true`，后2个返回 `false`
2. 最终 `supplier_apply_count` = 5
3. `recruit_list.list_status` 应为 25（已抢完），前提是系统有定时任务或回调更新状态
4. 第6个请求（无论哪个寄卖商）都会因竞争池已满返回失败

### 3.3 场景三：寄卖商撤回招募车

**操作说明**：寄卖商甲（supplier_id=1001）在招募车列表页点击"撤回招募车"。

**请求参数**：
```json
POST /scms/consignment/recruit/supplier/v1/cart/withdraw
{
  "recruitNo": "SC20260601001",
  "supplierId": 1001,
  "groupId": 2001
}
```

**后台处理流程**：
```
step 1: 根据 recruitNo 查找 recruit_list → id=1
        → 查询 recruit_apply: findByRecruitIdAndSupplierId(1, 1001)
        → 返回 apply_id=1, apply_status=10

step 2: 申请Redis锁 key="consignment:recruit:lock:1"

step 3: 物理删除申请记录（不依赖@TableLogic逻辑删除）
        → DELETE FROM scms_consignment_recruit_apply WHERE id=1

step 4: 竞争池降级（原子更新）
        → UPDATE scms_consignment_recruit_list
          SET supplier_apply_count = IFNULL(supplier_apply_count, 0) - 1
          WHERE id=1
        → supplier_apply_count: 3→2

step 5: 释放Redis锁
```

**数据走向**：

```
 recruit_apply                     recruit_list
 ┌──────────────────────┐         ┌──────────────────┐
 │ id=1 (DELETED)       │         │ id=1             │
 │ recruit_id=1         │──物理删除→│ supplier_apply   │
 │ supplier_id=1001     │         │  3→2             │
 │ apply_status=10      │         │                  │
 └──────────────────────┘         └──────────────────┘
```

**验证点**：
1. 接口返回 `{"value": true}`
2. `recruit_apply` 表中 id=1 的记录已物理删除（不复存在）
3. `recruit_list.supplier_apply_count` = 2（减少了1）
4. 再次调用同一个接口应返回"申请不存在"（已被物理删除）

### 3.4 场景四：并发撤销招募车

**操作说明**：模拟多个寄卖商同时撤销同一清单的招募车。

**模拟数据**：当前竞争池有3人（supplier_id=1001,1002,1003），3人同时发起撤回。

**后台处理**：
```
step 1: 3个请求各自申请Redis锁

step 2: 请求A（supplier_id=1001）成功获得锁
        → 物理删除 apply_id=1
        → UPDATE recruit_list SET supplier_apply_count=3-1=2

step 3: 请求B（supplier_id=1002）成功获得锁
        → 物理删除 apply_id=2
        → UPDATE recruit_list SET supplier_apply_count=2-1=1

step 4: 请求C（supplier_id=1003）成功获得锁
        → 物理删除 apply_id=3
        → UPDATE recruit_list SET supplier_apply_count=1-1=0
```

**验证点**：
1. 3个请求均返回 `true`
2. 最终竞争池 `supplier_apply_count` = 0
3. 该清单3条申请记录全部被物理删除

**并发安全关键**：
- Redis锁保证各请求串行执行，不会出现 `supplier_apply_count` 同时被多个线程读写
- 物理删除（`DELETE`）而非逻辑删除（`UPDATE is_deleted=1`），确保后续重新加入不会产生重复数据

### 3.5 场景五：开样品CE单

**操作说明**：寄卖商甲在招募车列表页，对清单SC20260601001点击"开样品CE单"。

**前置条件**：supplier_id=1001 已加入清单（apply_status=10），已重新加入。

**请求参数**：
```json
POST /scms/consignment/recruit/supplier/v1/cart/appendSku
{
  "recruitNoList": ["SC20260601001"],
  "supplierId": 1001,
  "groupId": 2001
}
```

**后台处理流程（核心 `createCeCore` 方法）**：
```
step 1: 根据 recruitNo 查询 recruit_list → id=1
        → findByRecruitIdAndSupplierId(1, 1001) → apply_id=新ID, status=10

step 2: 验证 apply_status=10（已加入）且 ce_bill_no=null（未开过CE）

step 3: 获取该清单全部SKU
        → SELECT * FROM scms_consignment_recruit_list_sku WHERE recruit_id=1
        → 返回30条SKU记录，包含 sku_id, cost_price

step 4: 收集所有 sku_id，查询样品信息
        → 调用 sampleSkuIdInfoService.getSampleSkuInfoMapBySkuIdList()
        → 获取每个SKU的 tempId 和 ux168Id

step 5: 创建CE主表
        → ceBillService.createCeMaster(supplierId, factoryId, ...)
        → 返回 ceBillNo="CE20260601001"

step 6: 为每个SKU创建CE明细
        → ceBillService.createCeDetails(ceBillNo, skuId, tempId, ux168Id, costPrice, ...)
        → 循环30次，逐条创建CE明细

step 7: 更新申请记录
        → UPDATE scms_consignment_recruit_apply SET
            apply_status=20(已开CE),
            ce_bill_no='CE20260601001',
            ce_create_time=NOW()
          WHERE id=apply_id
```

**数据走向**：

```
 recruit_apply                              CE主表(外部系统)
 ┌──────────────────────┐                 ┌──────────────────────┐
 │ apply_status: 10→20  │                 │ ce_bill_no:          │
 │ ce_bill_no: null→    │                 │   CE20260601001      │
 │   CE20260601001      │                 │ supplier_id: 1001    │
 │ ce_create_time: NOW  │                 │ factory_id: 101      │
 └──────────────────────┘                 └──────────────────────┘

 recruit_list_sku                         CE明细表(外部系统)
 ┌──────────────────────┐                ┌──────────────────────┐
 │ sku_id=5001 cost=25  │──→ createCe──→│ ce_bill_no:          │
 │ sku_id=5002 cost=30  │    Details     │   CE20260601001      │
 │ ...共30条            │  (循环30次)    │ sku_id: 5001         │
 └──────────────────────┘                │ temp_id: xxx         │
                                          │ cost: 25.00          │
                                          └──────────────────────┘
```

**验证点**：
1. 接口返回 `{"value": true}`
2. `recruit_apply` 表中该记录的 `apply_status` = 20（已开CE）
3. `recruit_apply.ce_bill_no` 有值
4. `recruit_apply.ce_create_time` 有值（当前时间）
5. CE系统中有对应的主表和明细记录

## 4. 并发测试专项

### 4.1 并发加入招募车（Redis锁保护）

**测试场景**：同一时刻，5个不同的寄卖商同时申请加入同一个招募清单（池容量5人，已有2人，剩余3个名额）。

**并发请求数**：5个平行请求。

**测试工具建议**：
- 使用 `curl` 脚本 + `&` 后台并行，或
- 使用 JMeter 线程组（5个线程同步触发），或
- 使用 Postman Runner 并发

**预期结果**：
- 前3个请求成功（竞争池 2→5）
- 后2个请求失败（竞争池已满返回false或抛出异常）

**实际验证步骤**：
1. 记录测试前 `supplier_apply_count` 的值
2. 发起5个并发请求
3. 查询接口返回，统计成功和失败数量
4. 查询 `recruit_list` 表确认 `supplier_apply_count` = 5
5. 查询 `recruit_apply` 表确认增加了3条新申请记录

### 4.2 并发撤销招募车

**测试场景**：3个已加入的寄卖商同时撤回同一个招募清单。

**并发请求数**：3个平行请求。

**预期结果**：
- 3个请求全部成功
- 竞争池从3降为0
- 3条申请记录全部被物理删除

**验证步骤**：
1. 确认3条申请记录在撤回前存在
2. 发起3个并发撤回请求
3. 确认所有请求返回 `true`
4. 查询 `recruit_apply` 表确认记录已硬删除
5. 查询 `recruit_list` 表确认 `supplier_apply_count` = 0

### 4.3 竞争池满员边界测试

**测试场景**：竞争池已有5人（已满），此时又有寄卖商尝试加入。

**前置条件**：`supplier_apply_count` = 5 或 清单状态为25（已抢完）

**请求参数**：
```json
{
  "recruitNo": "SC20260601001",
  "supplierId": 1009,
  "groupId": 2009
}
```

**预期结果**：
- 接口抛出业务异常："当前清单竞争池已满，无法加入"
- 或接口返回 `{"value": false}`
- `supplier_apply_count` 保持不变

## 5. 页面字段数据来源映射

| 清单SKU数量 | skuCount | 关联 `recruit_list` 表的 `sku_count` |
| 预计寄卖总价(元) | estimatedCost | 关联 `recruit_list` 表的 `estimated_cost` |

## 6. 异常场景与边界测试

### 6.1 重复加入检测
**操作**：寄卖商加入招募车成功后，立即再次调用加入接口。
**预期**：接口返回业务异常或false（不允许重复加入）。
**检查**：`recruit_apply` 表没有重复记录。

### 6.2 撤回后立即再加入
**操作**：寄卖商撤回招募车，然后立刻再次加入同一个清单。
**预期**：加入成功，因为撤回是物理删除，不会与旧记录冲突。
**检查**：`recruit_apply` 表中只有一条该寄卖商对该清单的新申请记录。

### 6.3 已开CE的申请不可重复开单
**操作**：对已开CE的申请记录再次调用开CE单接口。
**预期**：跳过该记录（批量模式）或返回失败消息（单ID模式）。
**检查**：`ce_bill_no` 不变，没有重复创建CE主表。

### 6.4 清单不存在/状态无效
**操作**：传入不存在的清单编号/ID，或清单状态不为招募中。
**预期**：返回失败，提示"清单不存在"或"状态不允许"。

### 6.5 竞争池降级不产生负数
**操作**：在竞争池为0时主动撤销招募车（极端情况）。
**预期**：`supplier_apply_count` 不会降为负数（`IFNULL(supplier_apply_count,0) - 1` 在MySQL中会保证最小为0）。
