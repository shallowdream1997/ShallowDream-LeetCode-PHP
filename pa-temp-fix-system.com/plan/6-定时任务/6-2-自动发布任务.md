# 6-2 自动发布任务 - AutoPublishJob

## 一、概述

| 项目 | 说明 |
|------|------|
| **调度频率** | 每周三 10:00（由 `publishDayOfWeek=3` 配置） |
| **XXL-Job Handler** | `consignmentRecruitAutoPublishJobHandler` |
| **Service** | `ConsignmentRecruitAutoPublishService` |
| **核心逻辑** | 将已组单完成的待发布清单，按规则自动发布为当前轮次的新一轮招募 |
| **触发方式** | XXL-Job 每周定时触发 / API 手动触发 (`triggerAutoPublish`) |
| **操作者** | `SYSTEM_AUTO_PUBLISH_OPERATOR` = `system(自动发布)` |

---

## 二、数据源

| 来源 | 表/配置 | 字段 | 说明 |
|------|---------|------|------|
| **上游依赖** | `recruit_list` (由AutoGroupJob产出) | 参考 [6-0](6-0-任务间依赖与防故障策略.md) | 需做防御性校验 |
| **读取** | `recruit_list` | `id, recruit_no, sku_count, create_time, group_time` | 筛选：`list_status=10(待发布)` |
| **读取** | `recruit_list_sku` | `sku_status` | 待更新：从 `20(已组单)` → `30(已发布)` |
| **读取** | `recruit_publish` | `publish_round, publish_status` | 计算轮次、检查活跃轮次 |
| **读取** | `CommonDictConfig` | `publishVisibleTime(10:00:00), applyStartTime(14:00:00), applyEndTime(21:00:00), dailyMaxPublishCount(20), publishDayOfWeek(3), holidayBlacklist` | 配置 |
| **计算** | per-recruit | `publish_round` | `MAX(非CANCELLED记录的publish_round) + 1` |
| **计算** | 系统 | `publish_end_time` | 当天 `23:59:59` |
| **写入** | `recruit_publish`（INSERT） | 全部发布相关字段 | 生成发布记录 |
| **写入** | `recruit_list`（UPDATE） | 状态 + 时间 + 操作人字段 | 更新清单主表 |
| **写入** | `recruit_list_sku`（UPDATE） | `sku_status=30(已发布)` | SKU标记为已发布 |
| **写入** | `action_log`（INSERT） | 操作日志 | action=publish |

## 四、publish_round 计算规则

```
publish_round = per-recruit MAX(非CANCELLED记录的 publish_round) + 1

示例：
  清单 SC2605150007 历史发布记录：
    轮次1: CANCELLED(已作废)  ← 不计入MAX
    轮次2: CANCELLED(已作废)  ← 不计入MAX
    轮次3: RECRUITING(招募中) ← MAX=3
  → 新发布轮次 = 3 + 1 = 4

注意：publish_round 是按清单维度递增的，不是 ISO 周数
```

## 五、状态走向

```
recruit_list:
  10(待发布) ─── AutoPublishJob ───→ 20(招募中)
  后续流转: 20 → [25] → 35(EvalStartJob) → 50(AutoAwardJob)

recruit_list_sku:
  20(已组单) ────→ 30(已发布)

recruit_publish:
  INSERT → publish_status = 20(招募中)
  旧轮次 → publish_status = 100(作废/CANCELLED)
```

## 六、每日上限控制

```
countTodayPublished():
  统计今日已发布（非CANCELLED）的 publish 记录数
  状态范围: 10/20/25/30/35/50/60（排除100-CANCELLED）

remainingQuota = dailyMaxPublishCount(20) - countTodayPublished()
  如果剩余额度 <= 0 → 跳过执行
  如果待发布数 > 剩余额度 → 截取前N条
```

## 七、回滚发布功能

`rollbackPublish(List<String> recruitNos)` — 将招募中的清单回滚到待发布状态：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 查询 `list_status=20` 的目标清单 | 根据 recruitNos 匹配 |
| 2 | 物理删除 `recruit_publish` 记录 | `physicalDeleteByRecruitIds` |
| 3 | 主表回滚：`list_status → 10`，清空发布字段 | `publishBy/publishBeginTime/applyBeginTime/applyEndTime/publishEndTime → null` |
| 4 | SKU 回滚：`sku_status 30 → 20(GROUPED)` | 仅回滚已发布的SKU |
| 5 | INSERT action_log | `content="回滚发布 - 清单状态恢复为待发布"` |

API 入口：`POST /v1/rollbackPublish` — 传入招募编号列表

## 八、表数据处理

| 操作 | 表 | SQL/说明 |
|------|-----|----------|
| SELECT | `recruit_list` | `WHERE list_status=10`，排序+上限截取 |
| SELECT | `recruit_publish` | `WHERE recruit_id=?` 查已有轮次 |
| UPDATE | `recruit_publish` | 旧轮次 `publish_status=100(CANCELLED)` |
| INSERT | `recruit_publish` | 每张清单一条发布记录 |
| UPDATE | `recruit_list` | `batchUpdateStatus(id, 10, 20)` 乐观锁 |
| UPDATE | `recruit_list_sku` | `sku_status=30` |
| INSERT | `action_log` | 每条清单一条日志 |

## 九、难点与解决点

| 难点 | 解决 |
|------|------|
| **手动发布和自动发布并发** | 使用 `batchUpdateStatus(id, WAIT_PUBLISH, RECRUITING)` 乐观锁：影响行数=0说明已被手动发布 |
| **清单已有活跃轮次** | 查询 `recruit_publish` 判断是否有活跃状态（20/25/30/35/50/60），有则跳过 |
| **publish_round 唯一键冲突** | per-recruit MAX+1 递增，作废旧轮次后插入新轮次 |
| **周三时间校验** | 通过配置 `publishDayOfWeek=3` 校验 + `holidayBlacklist` 节假日黑名单 |
| **10:00可见与14:00可抢之间的"只可见不可抢"** | `publish_begin_time` 控制前端列表可见性，`apply_begin_time` 控制加入按钮可用性 |
| **每日发布上限** | `dailyMaxPublishCount` + `countTodayPublished()` 动态计算剩余额度 |

## 十、防故障策略（针对AutoGroupJob上游依赖）

> **依赖关系**: AutoGroupJob（另一研发）INSERT `recruit_list` → AutoPublishJob 读取并发布
> **详细策略**: 参见 [6-0-任务间依赖与防故障策略.md](6-0-任务间依赖与防故障策略.md)

| 措施 | 说明 |
|------|------|
| **枚举常量** | 所有状态值使用 `枚举类.getCode()`，不硬编码数字 |
| **防御性校验** | 对上游数据做字段完整性检查（id/skuCount/groupTime） |
| **乐观锁UPDATE** | `batchUpdateStatus(id, 10, 20)` 保障只操作正确状态的清单 |
| **空数据容忍** | 无待发布清单时记录INFO日志正常结束，不抛异常 |
| **逐条异常隔离** | 单条清单发布失败不影响其他清单的处理 |
| **执行监控** | 每次执行记录总数/成功/失败数量 |
