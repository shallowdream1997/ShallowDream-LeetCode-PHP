# 6-9 评选开始任务 - EvalStartJob

## 一、概述

| 项目 | 说明 |
|------|------|
| **调度频率** | 每天 00:05（合并原CeTimeoutClean 03:00，同步执行超时清理） |
| **XXL-Job Handler** | `consignmentRecruitEvalStartJobHandler` |
| **Service** | `ConsignmentRecruitEvalStartService` |
| **核心逻辑** | Step 0 全局清理超时apply(10→90)；Step 1~2 将 `publishEndTime` 已到期的清单转为 **35(评选中)**，同步将已开CE单(20)的apply→**30(等待评选)**，**开CE单成功即可等待评选，无需等质检** |
| **操作者** | `SYSTEM_EVAL_START_OPERATOR` = `system(评选开始)` |

---

## 二、数据源

| 操作 | 表/配置 | 字段 | 说明 |
|------|---------|------|------|
| **读取** | `recruit_list` | `id, list_status, publish_end_time, award_time` | 筛选可进入评选的清单 |
| **读取** | `recruit_publish` | `publish_status` | 同步更新发布轮次状态 |
| **读取** | `recruit_apply` | `id, apply_status, ce_bill_no` | 同步更新招募车状态 |
| **写入** | `recruit_list` | `list_status → 35` | 乐观锁更新清单状态 |
| **写入** | `recruit_publish` | `publish_status → 35` | 同步更新活跃发布轮次 |
| **写入** | `recruit_apply` | `apply_status: 20(已开CE)→30(等待评选)` | 同步招募车，无需等质检 |
| **写入** | `recruit_apply` | `apply_status: 10(已加入超时)→90(超时清出)` | 合并超时清理 |
| **写入** | `action_log` | `action=eval_start / timeout_clean` | 操作日志 |

## 四、状态走向

```
recruit_list:
  20(招募中) ─── EvalStartJob ───→ 35(评选中)
  25(已抢完) ─── EvalStartJob ───→ 35(评选中)

recruit_publish:
  20(招募中) / 25(已抢完) ───→ 35(评选中)

recruit_apply (同步更新):
  10(已加入,超时) ───→ 90(超时清出) ← Step 0 合并超时清理
  20(已开CE) ───→ 30(等待评选) ← 开CE单即可，无需等质检
  30(已等待评选) ───→ 不变（已是目标状态）
  10(已加入,未超时) ───→ 不变（等开CE单，不受EvalStart影响）

完整链路：
  10(待发布) → 20(招募中) → [25(已抢完)] → 35(评选中) → 50(分配中) → 60(清单完成)
```

## 五、表数据处理

| 操作 | 表 | 说明 |
|------|-----|------|
| SELECT | `recruit_list` | `WHERE list_status IN (20,25) AND publish_end_time <= NOW()` |
| SELECT | `recruit_apply` | `findById(recruitId)` 查申请列表 + `findTimeoutApplies` 查超时 |
| UPDATE | `recruit_list` | `batchUpdateStatus(id, oldStatus, 35)` 乐观锁 |
| UPDATE | `recruit_publish` | 活跃轮次 `publish_status → 35` |
| UPDATE | `recruit_apply` | step0: `status=10超时→90` step2c: `status=20→30` |
| INSERT | `action_log` | `action=eval_start / timeout_clean` |

## 六、与其他Job的关系

| 关系 | Job | 说明 |
|------|-----|------|
| **上游** | AutoPublishJob (6-2) | 发布时设置 `publish_end_time`（当天23:59:59），EvalStartJob 在次日凌晨扫描 |
| **上游** | 其他apply创建 | apply在发布期间创建(10→20)，EvalStart 处理超时和已开CE的 apply |
| **下游** | AutoAwardJob (6-4) | 查询条件包含 `list_status=35`，且 apply_status=30 的参与评选 |
| **下游** | CeArrivalCheckJob (6-10) | 清单进入35后，CE到货质检仍在进行，仅查status=30的apply |

## 七、难点与解决点

| 难点 | 解决 |
|------|------|
| **清单与apply的状态一致性问题** | Step2c `syncApplyStatusToWaitAward()` 同步已开CE单的apply→30，保障一致性 |
| **不影响覆盖率/同源加权** | 只改 `apply_status`，覆盖率/同源加权在 CeArrivalCheck 中计算 |
| **与 CeArrivalCheck 无竞态冲突** | CeArrivalCheck 仅处理 status=30 的apply，EvalStart 改20→30后交由CeArrivalCheck |
| **publishEndTime 为前一天23:59:59，次日00:05扫描** | 使用 `publishEndTimeLe(new Date())` 比较 |
| **乐观锁并发** | `batchUpdateStatus(id, oldStatus, EVALUATING)` 使用清单当前状态作为乐观锁条件 |
| **已评选清单不重复处理** | Java 过滤 `award_time IS NULL` |
| **publish轮次可能多条** | 只更新状态为 RECRUITING/FULL_SNAPPED 的活跃轮次 |
| **空数据容忍** | 无可进入评选的清单时记录INFO日志正常结束 |
