# 6-5 CE单超时清出任务 - 已合并到评选开始任务(6-9)

> **合并说明**：CeTimeoutCleanJob 已合并到 EvalStartJob（6-9）。
> EvalStartJob (00:05) 在执行评选开始前，会先全局清理超时未开CE单的申请。
> 详见 [6-9-评选开始任务.md](6-9-评选开始任务.md) 的 Step 0。

**原逻辑**：将加入招募车后超过配置天数（默认3天）仍未开CE单的申请（status=10→90）自动清出。
**原调度时间**：每天 03:00
| **操作者** | `SYSTEM_CE_TIMEOUT_CLEAN_OPERATOR` = `system(CE超时清出)` |

---

## 二、数据源

| 操作 | 表/配置 | 字段 | 说明 |
|------|---------|------|------|
| **读取** | `recruit_apply` | `id, recruit_id, supplier_id, create_time` | 筛选超时未开单的申请 |
| **读取** | `CommonDictConfig` | `ceCreateTimeoutDays(3)` | 超时天数配置 |
| **更新** | `recruit_apply` | `apply_status=90, removed_type=timeout, cancel_reason` | 标记为超时清出 |
| **写入** | `action_log` | `action=timeout_clean` | 操作日志 |

---


## 五、表数据处理

| 操作 | 表 | 说明 |
|------|-----|------|
| SELECT | `recruit_apply` | `WHERE apply_status=10 AND create_time < deadline` |
| UPDATE | `recruit_apply` | `SET apply_status=90, removed_type='timeout', cancel_reason='超时未开CE单，系统自动清出'` |
| INSERT | `action_log` | `action=timeout_clean` |

---

## 六、难点与解决点

| 难点 | 解决 |
|------|------|
| **超时时间计算规则** | 从加入的**第二天凌晨**起算，而非加入时间+72小时。实现：`当前时间 - (ceCreateTimeoutDays + 1)天` 的 `23:59:59` |
| **并发清出时数据竞争** | `apply_status=10` 条件保障，影响行数=0说明已被其他任务处理 |
| **清出后竞争池人数减少** | 本Job暂时不处理list_status变化（满5人变招募中等场景） |
| **配置动态化** | `ceCreateTimeoutDays` 从 `CommonDictConfig` 读取，支持动态调整 |
